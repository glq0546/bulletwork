# 技术攻坚记录

**日期**：2026-06-13

## 核心内容

本文件记录项目中已经解决的技术问题、耗时和解决方案。重点不是炫技，而是避免以后重复踩坑。

## 一、API端点404

现象：前端调用API时出现404。原因是前端URL和Worker实际路由不一致。解决方案：将前端URL从workers.dev相关地址调整为自定义API域名 api.glq-api.asia。教训：前端、Worker路由、自定义域名三处必须同步。

## 二、DNS污染

现象：ISP对workers.dev解析到错误IP，导致访问异常。耗时数小时以上。解决方案：使用自定义域名，Cloudflare开启橙色代理，配置Worker路由。教训：workers.dev不稳定时，不要死磕，直接上自定义域名。

## 三、Worker 522错误

现象：访问api.glq-api.asia出现522。原因：Worker路由未正确绑定。解决方案：为api.glq-api.asia添加Worker路由。

## 四、Vercel部署不生效

现象：修改代码后线上长期不更新。耗时约7天反复排查。解决方案：断开GitHub连接，改为本地直接部署。部署命令：`vercel --prod --yes`。教训：Vercel自动部署链路异常时，先用本地部署确认，不要在GitHub联动里绕圈。

## 五、CSP修复

现象：PayPal、API或页面资源被Content Security Policy拦截。解决方案：在vercel.json中更新CSP策略；加入PayPal SDK白名单；修复Worker路由；同步API Key更换。教训：CSP、安全头、第三方SDK必须一起检查。

## 六、DeepSeek API Key失效

现象：AI调用失败。解决方案：更换新Key，并同步更新三处相关配置。安全规则：API Key只放本地配置或平台环境变量，不写入数据库和公开文件。

## 七、vercel.json配置冲突

现象：解码项目配置覆盖子弹项目配置，导致首页404。原因：两个项目共用同一repo，vercel.json路由/build配置相互影响。解决方案：移除或修复冲突的builds/routes限制，恢复首页访问。教训：多项目共仓时，配置文件是高风险点，部署前必须检查主站和子目录。

## 八、首页404

现象：主站首页访问异常。解决方案：调整vercel.json，去掉导致静态首页无法命中的限制。

## 九、PayPal沙盒Plan不互通

现象：PayPal沙盒计划和真实环境不互通。解决方案：放弃沙盒互通排查，改用真实付费测试方案。

## 十、克劳扣得覆盖润色版文章

现象：已经由谷歌AI润色的文章被克劳扣得修改覆盖。解决方案：修改前必须备份；内容生成和代码部署分工明确。

## 十一、WARP连不上

现象：网络工具不可用。解决方案：改用libcyberdesktop。

## 十二、最终原则

技术问题先确认链路：前端 → 路由 → Worker → API → 第三方。部署问题先确认线上是否真的更新。配置文件修改前备份。敏感Key不进入文档库。解决方案要沉淀成SOP，而不是只靠记忆。
