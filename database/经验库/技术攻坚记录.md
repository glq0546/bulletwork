# 技术攻坚记录

**日期**：2026-06-13

## 核心内容

本文件记录项目中已经解决的技术问题、耗时和解决方案。重点不是炫技，而是避免以后重复踩坑。

## 一、API端点404

现象：前端调用API时出现404。原因是前端URL和Worker实际路由不一致。解决方案：将前端URL从workers.dev相关地址调整为自定义API域名 api.glq-api.asia。教训：前端、Worker路由、自定义域名三处必须同步。

## 二、DNS污染

现象：ISP对workers.dev解析到错误IP，导致访问异常。耗时数小时以上。解决方案：使用自定义域名，Cloudflare开启橙色代理，配置Worker路由。教训：workers.dev不稳定时，不要死磕，直接上自定义域名。

## 三、Worker 522错误

现象：访问api.glq-api.asia出现522。原因：Worker路由未正确绑定。解决方案：为api.glq-api.asia添加Worker路由。

## 四、Vercel部署不生效

现象：修改代码后线上长期不更新。耗时约7天反复排查。解决方案：断开GitHub连接，改为本地直接部署。部署命令：`vercel --prod --yes`。教训：Vercel自动部署链路异常时，先用本地部署确认，不要在GitHub联动里绕圈。

## 五、CSP修复

现象：PayPal、API或页面资源被Content Security Policy拦截。解决方案：在vercel.json中更新CSP策略；加入PayPal SDK白名单；修复Worker路由；同步API Key更换。教训：CSP、安全头、第三方SDK必须一起检查。

## 六、DeepSeek API Key失效

现象：AI调用失败。解决方案：更换新Key，并同步更新三处相关配置。安全规则：API Key只放本地配置或平台环境变量，不写入数据库和公开文件。

## 七、vercel.json配置冲突

现象：解码项目配置覆盖子弹项目配置，导致首页404。原因：两个项目共用同一repo，vercel.json路由/build配置相互影响。解决方案：移除或修复冲突的builds/routes限制，恢复首页访问。教训：多项目共仓时，配置文件是高风险点，部署前必须检查主站和子目录。

## 八、首页404

现象：主站首页访问异常。解决方案：调整vercel.json，去掉导致静态首页无法命中的限制。

## 九、PayPal沙盒Plan不互通

现象：PayPal沙盒计划和真实环境不互通。解决方案：放弃沙盒互通排查，改用真实付费测试方案。

## 十、克劳扣得覆盖润色版文章

现象：已经由谷歌AI润色的文章被克劳扣得修改覆盖。解决方案：修改前必须备份；内容生成和代码部署分工明确。

## 十一、WARP连不上

现象：网络工具不可用。解决方案：改用libcyberdesktop。

## 十二、Pandoc代码块截断问题与修复

现象：生成PDF或HTML时，Pandoc在处理长代码块、长模板、英文争议信内容时出现截断、换行错乱或后半段丢失，导致最终PDF内容不完整。

原因：代码块内存在过长行、特殊字符、未闭合围栏、HTML转义字符或Markdown列表嵌套过深。Pandoc对复杂Markdown结构容错有限，尤其是长模板信件和表格混排时容易出问题。

修复：将长代码块改为普通引用或分段文本；模板信件不要全部放在一个超长代码围栏内；每个模板单独检查开头和结尾；生成后用文本搜索核对关键句是否完整；PDF发布前必须抽查每个模块最后一段。

教训：PDF生成不是写完Markdown就结束，必须检查最终成品。长模板、表格、代码块是Pandoc转换高风险区域。

## 十三、克劳扣得自动更新导致程序文件损坏

现象：克劳扣得或相关CLI工具自动更新后，程序文件出现异常，导致命令无法正常运行或行为和之前不一致。

处理流程：先确认是否刚发生自动更新；检查版本号；不要立刻继续执行重要写入任务；必要时重启终端；保留当前任务文件；如果工具行为异常，改用手动文件操作或等待稳定后再继续。

教训：自动更新本身也可能成为故障源。关键部署、PDF生成、批量改文件前，尽量避免在工具刚更新后马上执行不可逆操作。

## 十四、痛点雷达多渠道配置经验

现象：痛点雷达从单一方向扩展到多渠道、多关键词后，容易出现搜索结果噪音高、方向混杂、Token消耗过大、某些方向无数据却被静默跳过的问题。

经验：方向必须先分组，再配置关键词和板块；每个方向每次最多提炼前10条；过滤促销、品牌广告、纯新闻和无文字链接；搜索结果为零时也要在简报中标注；结构化字段必须固定，包括painPoint、userPersona、emotionalIntensity、willingnessToPay、sourceUrl、evidence、direction。

教训：痛点雷达的价值不在抓得多，而在抓得准、可复盘、可产品化。多渠道配置必须有统一输出格式，否则后续无法变成数据资产。

## 十五、G2产品上架完整流程与关键卡点

流程：注册/登录G2账号；提交产品基本信息；填写官网、分类、描述、Logo、截图；补充公司信息和联系邮箱；等待G2审核；根据进度条补齐缺失项；后续需要真实用户评价补完可信度。

关键卡点：产品页面可以创建，但完整度不一定达到发布/展示要求；G2对评价和公司信息有要求；早期产品没有用户评价时进度会卡住；外链和品牌信息要和官网一致。

教训：G2适合作为信任背书，不适合作为早期主要获客渠道。它需要真实用户评价，应该等产品有用户后再投入更多精力完善。

## 十六、最终原则

技术问题先确认链路：前端 → 路由 → Worker → API → 第三方。部署问题先确认线上是否真的更新。配置文件修改前备份。敏感Key不进入文档库。解决方案要沉淀成SOP，而不是只靠记忆。
